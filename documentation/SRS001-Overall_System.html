Introduction

Purpose

This document specifies the general requirements of the whole Ecology Sim system. This document regards the software at the highest level of abstraction, considering the general classes of users who will use the system, the functionality they would request, and the resulting flow of data throughout the software's subsystems necessary to accomplish that functionality. This document thus also serves as a catalog of subsystems, each of which requiring their own specification document.

The intended audience of this document is, like all documents produced as a part of the Ecology Sim project, primarily myself. While I certainly will reference them in the production of code, the primary reason these documents exist is that the act of producing them with such rigor guides me in the design process. Without them I would be left to aimlessly write code that lacks the cohesion to achieve my goals with this project.

A secondary audience to this document is that of any interested contributors to the Ecology Sim project. A solid foundation of specific requirements should help streamline the contribution process.

The third audience considered in the production of this document is that of any end user who wishes to gain a better understanding of how the system functions.

Scope

Ecology Sim is a software system that simulates a natural environment containing plants and animals. The software simulates the inner subprocesses making up each simulation entity as well as the interactions between entities. The software also tracks changes in the dynamic properties of the simulation and its constituent entities and displays those changes when prompted by the user.

This software is not designed for large scale simulations. The size of the simulated environment and number of simulated entities are limited by design. Focus instead is placed on the design of sufficiently complex and varied subsystems such that the interactions between them are more interesting.

The simulation generated by the software is:

  • Stochastic: The behavior of simulated creatures depends partially on randomly generated numbers.
  • Dynamic: The target of study is the way in which the simulated environment and creatures therein change over time.
  • Temporally Continuous: Updates occur in fixed time steps.
  • Spatially Discrete: Objects in the simulation are placed in the cells of a regular grid.

The Ecology Sim software is intended for use by ALife hobbyists and not for serious research. The dynamics involved and design of subsystems are not derived from first principles but instead from higher level abstractions. The goal of the software is to produce interesting interactions.

Terms, Acronyms, and Abbreviations

Table of Definitions of Terms Used in Document
Term Definition
Alife

Short for Artificial Life. The field of Alife seeks to understand the processes governing and the processes emerging from natural living systems by means of simulation in or implementation in an artificial substrate.

In the field there are two stances regarding whether or not Alife simulations and implementations can be considered to be truly alive in the same way that natural life is.

  • The Strong Alife position considers life to be a process that can be abstracted away from a certain medium, that so long as the fundamental processes governing living systems are present in a given system it may be considered "alive".
  • The Weak Alife position denies that Artificial Life is truly alive but instead considers it a model for natural living systems. Like any other model, Alife's main purpose in this stance is to better understand the processes underlying natural living systems.

NOTE: This project assumes the Weak position for the purposes of design

Dialog

A Dialog is a Window (a Widget not embedded in a parent) that is used for short-term interaction with the user. Typical applications include data entry, warning or error prompts, and about boxes giving information about an application.

Dialogs may be modal or modeless. The former prevents the user from interacting with the main application until Dialog interaction has been completed and applied, whereas the latter allows the user to freely interact with either.

Entity

Entities are the general class encompassing all the agents to be simulated by Ecology Sim. This includes both plants and animals. Each of those categories may be further subdivided to yield more specialized artificial organisms.

Environment

The Environment is the spatial and temporal artificial landscape within which Entities interact. The Environment is composed of a regular 2d grid of square cells. Each cell is considered to be adjacent to the cells to the top, bottom, left, right, and all four diagonals to itself (equivalent to a Moore neighborhood in Cellular Automata).

Environments have properties that can be global, affecting every cell, such as temperature and light intensity, or local, affecting only certain cells, such as terrain type and soil moisture level.

Each cell in an Environment has contents which can include one or more Entities and/or Items.

Item

An Item is any inert object simulated by Ecology Sim. That is, an Item is an object which does not initiate interactions with any other object. Examples include: rocks and dead Entities.

Layout

In Qt the Layout system provides a Widget a means of automatically arranging their child Widgets within their available space.

For more information, visit the Qt Documentation page.

Model

A Model is one part of the Model-View-Controller design pattern, simplified in Qt to just the Model-View pattern. A Model is not the same as the underlying data but instead acts as a common interface to that data based on model indexes.

For more information, visit the Qt Documentation page.

UML Class Diagram

Class Diagrams are tools used in the Unified Modeling Language (UML) to visually depict the structure of a system by means of enumerating their classes (using the Object Oriented programming paradigm) and the relationships between those classes.

For a detailed explanation of the features of UML Class Diagrams and their meanings, see the IBM Developer article on them.

View

A View is one part of the Model-View-Controller design pattern, simplified in Qt to just the Model-View pattern. Views are in charge of retrieving data via Model indexes and presenting that data to the user. Views can display underlying data in a variety of ways depending on the View's implementation.

For more information, visit the Qt Documentation page.

Widget

Widgets are the fundamental building blocks of GUIs in Qt. Primitive Widgets such as buttons and scroll bars act as basic units of interaction between the user and the program. Composite Widgets contain other Widgets organized by a Layout. A Widget not embedded in a parent Widget is a Window.

Document Overview

The next section, General Description, provides a general overview of the Ecology Sim application, focusing on interactions at the outermost border of the system as well as providing an interface for further requirements on the subsystems. This section doesn't state any specific requirements but rather provides background for those requirements. The Layout system is described and the core user interactions with the GUI's menu functions are laid out. Potential users are grouped based on interest and expertise and those user groups are associated with their most used functions to determine use frequency. The simulation model is briefly described and any topics deep enough for further discussion of requirements are delegated to their own documents.

The section Specific Requirements then delves into the specific functional requirements governing this level of the system. Each menu function is fully described along with their associated preconditions, input testing, response, etc. The functionality of the Layout system is also described, as well as the requirements for the Widgets therein. Then the highest level requirements to the Model data structures are determined. Additional requirements relating to performance, design constraints, and other requirements are described.

Finally, the System Architecture section presents a high-level overview of the anticipated system archictecture using UML Class Diagrams to be later refined in subsequent documentation.

Product Perspective

System Interfaces

As this document describes the Ecology Sim software at the highest level, the only system the software interacts with is the User's operating system.

Within the Ecology_Sim directory are three subdirectories: documentation, examples, and saves. The inner structure of the documentation directory shall be specified further in a later document [TODO reference document]. The examples directory contains subdirectories entities and environments as well as example simulation files. These contain example entity files and example environment files respectively. The saves directory is set aside for the user's saved data. Further organization of this directory is subject to the user's wants and needs.

Note that this is subject to change upon any revisions to this document.

User Classes and Characteristics

Though these software requirements are being set out far in advance of any functional prototype being available to users, the following potential user classes have been identified:

  • Casual User
  • Simulation Architect
  • Entity Designer
  • Long-Term Experimenter

These user classes are listed in rough order of importance. The foremost class is how every user starts out, as a Casual User. Thus the accessibility of the core functionality to that class is most important. The latter classes deal with progressively more nuanced functionality. Often, the requirements pertaining to the former classes also apply to the latter classes. An Entity Designer requires the same functionality as a casual user plus additional functionality.

Casual User

This is the class of users who interact with Ecology Sim on mostly a surface level. A Casual User may open the application, load a template Layout, set up an example Simulation with an example environment and example entities. They may watch the Simulation play out for a few minutes. They may leave it running and return to it later to see how things have evolved.

A Casual User doesn't delve into the subsystems comprising the Simulation, but only passively views the interactions of the running simulation.

Simulation Architect

This is the class of users that, along with interacting with the surface level functionalities of Ecology Sim, also is interested in the design of Simulations and Environments. A Simulation Architect may use the Environment Editor Widget to create new and interesting settings to see how a Simulation would evolve with it. They may edit the Simulator settings to suit their needs. They also are adept at using the keyboard controls of the Simulator View Widget.

Entity Designer

This is the class of users that, along with interacting with the surface level functionalities of Ecology Sim, also is interested in the design of Entities. An Entity Designer may use the Entity Selector Widget to deeply examine the underlying properties and systems comprising an Entity in the Simulation. They may use the Entity Editor Widget to design custom Entities to see how they fare in Simulations.

Operating Environment

Ecology Sim is software primarily developed for a Linux OS given that the current sole developer works on an Ubuntu 22.04.1 LTS system at the time of this document's drafting. However, Qt is a cross-platform framework. Thus, it isn't unreasonable to expect development to, at some point, expand to target Windows and Mac systems. Note that this expansion would only take place after the core functionality of the application has been established.
